# 第 7 集：调度、监听和选择器

## 本集目标

这一集进入进阶 API，但目标仍然很实际：让 UI 更新更可控。

我们会讲：

- `createWatch`：监听变化，并拿到新值和旧值。
- `createSelector`：处理列表选中态。
- `batch`：把多次更新合并。
- `untrack`：读取但不建立依赖。
- `flushSync`：需要立刻拿到 DOM 结果时同步刷新。
- `startTransition`：把不紧急的大更新放到低优先级。

## 开场口播

前面我们已经能写基本 UI 了。

但真实项目里还会遇到一些细节问题：

输入框变化后要发请求，但首次渲染不想发。

点列表项时，只希望旧选中和新选中的两项更新。

重置表单时，一次要改十几个字段，不想每改一个都触发一轮更新。

这类问题就属于“调度和监听”。

## 1. createWatch：监听某个值变化

### 口播

`createEffect` 是“读到什么就依赖什么”。

但有时我们只想明确监听某一个值，并且想拿到变化前和变化后的值。

这时用 `createWatch` 更直接。

### 代码

```js
const { createSignal, createWatch } = signal;

const [keyword, setKeyword] = createSignal('');

createWatch(
  keyword,
  (next, prev) => {
    console.log('keyword changed:', prev, '=>', next);
  },
  { defer: true }
);
```

### 讲解

`keyword` 是监听来源。

回调里的 `next` 是新值，`prev` 是旧值。

`defer: true` 表示首次创建时不执行回调，只在后续变化时执行。

这个选项在搜索、筛选、保存草稿时很常用。

## 2. 监听多个值

### 代码

```js
const [page, setPage] = createSignal(1);
const [pageSize, setPageSize] = createSignal(20);

createWatch(
  [page, pageSize],
  ([nextPage, nextPageSize], previous) => {
    loadList({
      page: nextPage,
      pageSize: nextPageSize,
    });
  },
  { defer: true }
);
```

### 讲解

当 source 是数组时，回调拿到的新值也是数组。

适合分页、排序、筛选条件共同决定一个请求的场景。

## 3. watch 和 effect 怎么选

### 口播

如果你是在描述“依赖变化后做一件副作用”，用 `createEffect`。

如果你明确要监听某几个来源，并且需要新旧值，用 `createWatch`。

### 对比

```js
createEffect(() => {
  document.title = keyword();
});
```

```js
createWatch(keyword, (next, prev) => {
  console.log(prev, next);
});
```

## 4. createSelector：列表选中态

### 口播

很多 UI 都有“当前选中项”：tab、菜单、表格行、商品规格。

最直接的写法是每一项都判断 `selectedId() === item.id`。

`createSelector` 可以把这个判断封装成一个函数，让代码更清楚。

### 代码

```js
const { createSignal, createSelector, For, render, jsx } = signal;

const [selectedId, setSelectedId] = createSignal('home');
const isSelected = createSelector(selectedId);

const tabs = [
  { id: 'home', label: '首页' },
  { id: 'order', label: '订单' },
  { id: 'user', label: '用户' },
];

render(
  () => jsx`
    <nav>
      ${For({
        each: () => tabs,
        key: (tab) => tab.id,
        children: (tab) => jsx`
          <button
            class=${() => (isSelected(tab().id) ? 'active' : '')}
            onClick=${() => setSelectedId(tab().id)}
          >
            ${() => tab().label}
          </button>
        `,
      })}
    </nav>
  `,
  app
);
```

### 讲解

`isSelected('home')` 返回当前是否选中了 home。

它适合所有“一个当前值，对比很多 key”的场景。

## 5. batch：多次修改合并成一次

### 口播

如果一次用户操作会修改多个状态，可以用 `batch` 包起来。

它的意义不是让代码更短，而是让更新作为一个整体发生。

### 代码

```js
const { batch } = signal;

function resetForm() {
  batch(() => {
    form.name = '';
    form.price = 0;
    form.stock = 0;
    form.published = false;
  });
}
```

### 讲解

没有 `batch` 时，每次写入都可能通知依赖。

有 `batch` 时，里面多次修改会等到最外层 `batch` 结束后再统一刷新。

适合表单重置、批量修改列表、导入数据后整体替换状态。

## 6. untrack：读取但不订阅

### 口播

响应式系统默认是“读了谁，就依赖谁”。

但有时我们只是临时读取一个辅助值，不希望它以后触发 effect 重跑。

这时用 `untrack`。

### 代码

```js
const { createEffect, untrack } = signal;

createEffect(() => {
  const id = selectedId();

  const cacheSnapshot = untrack(() => snapshot(cache));

  console.log('selected changed:', id, cacheSnapshot);
});
```

### 讲解

这个 effect 只依赖 `selectedId()`。

虽然里面也读了 `cache`，但因为放在 `untrack` 里，所以 `cache` 变化不会让这个 effect 重跑。

## 7. flushSync：同步刷新后读取 DOM

### 口播

大多数时候，我们不需要关心更新什么时候真正刷新。

但有些场景例外：比如打开弹窗后，马上要测量它的高度或位置。

这时可以用 `flushSync`。

### 代码

```js
const { createSignal, flushSync } = signal;

const [open, setOpen] = createSignal(false);
let panel;

function openAndMeasure() {
  flushSync(() => {
    setOpen(true);
  });

  const rect = panel.getBoundingClientRect();
  console.log(rect.height);
}
```

### 讲解

`flushSync` 会先执行里面的更新，然后立刻刷新普通 effect 队列。

所以它适合“更新状态后马上读 DOM”的少数场景。

不要把它当作默认写法。默认情况下，让运行时自己调度更新就可以。

## 8. startTransition：低优先级更新

### 口播

有些更新很重，但不需要阻塞用户当前操作。

例如输入关键词时，大列表筛选可能会比较慢。我们希望输入框先流畅响应，列表稍后更新。

可以用 `startTransition`。

### 代码

```js
const { createDeepStore, startTransition } = signal;

const state = createDeepStore({
  input: '',
  keyword: '',
});

function handleInput(event) {
  const value = event.currentTarget.value;

  state.input = value;

  startTransition(() => {
    state.keyword = value;
  });
}
```

### 讲解

`state.input` 是立即更新的，用来保证输入框体验。

`state.keyword` 是低优先级更新，用来触发大列表筛选。

这类优化一般等你真的遇到卡顿再用，不需要一开始就到处加。

## 9. 小项目片段：可筛选大列表

### 代码

```js
const state = createDeepStore({
  input: '',
  keyword: '',
  selectedId: null,
  rows: Array.from({ length: 2000 }, (_, index) => ({
    id: `row-${index}`,
    name: `商品 ${index}`,
  })),
});

const isSelected = createSelector(() => state.selectedId);

const visibleRows = createMemo(() => {
  const keyword = state.keyword.trim();
  if (!keyword) return state.rows;
  return state.rows.filter((row) => row.name.includes(keyword));
});

function onInput(event) {
  const value = event.currentTarget.value;

  state.input = value;

  startTransition(() => {
    state.keyword = value;
  });
}
```

### 讲解

这个片段里：

输入框值立即更新。

列表筛选低优先级更新。

选中态用 `createSelector` 判断。

这就是调度 API 和业务 UI 的结合方式。

## 本集结尾

这一集我们学会了：

- `createWatch` 适合明确监听变化，并拿到新旧值。
- `createSelector` 适合选中态。
- `batch` 合并多次更新。
- `untrack` 读取但不订阅。
- `flushSync` 用在更新后立刻读 DOM 的场景。
- `startTransition` 用在不紧急的大更新。

这些 API 不一定每天都用，但它们能让复杂 UI 更稳定、更可控。
